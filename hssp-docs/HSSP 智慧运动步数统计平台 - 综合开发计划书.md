# HSSP 智慧运动步数统计平台 - 综合开发计划书

| **维度**     | **详细说明**                                                 |
| ------------ | ------------------------------------------------------------ |
| **项目全称** | HarmonyOS & Smart Step Platform (HSSP)                       |
| **核心技术** | SpringBoot 2.7, MyBatis-Plus, Redis, Vue 3, ArkTS (HarmonyOS) |
| **架构参考** | 黑马头条 (Heima Leadnews) 多模块分层架构                     |
| **团队构成** | 组长(后端A-管理端) + 后端B(用户/步数) + 后端C(模型/积分) + Web端(D) + 鸿蒙端(E) |

------

## 一、 团队角色与垂直分工 (Matrix Responsibility)

为了实现 **Monorepo** 模式下的零冲突协作，各成员负责的物理文件夹及逻辑模块划分如下：

### 1. 后端架构与管理端 (组长：后端A)

- **负责文件夹**：`/hssp-backend/hssp-common` 与 `hssp-service` (Admin子包)
- **核心职责**：
  - **脚手架搭建**：配置 Maven 父工程及公共工具类（Result、Exception、JWT）。
  - **管理端逻辑**：实现积分兑换比例配置、商品上下架、用户状态管控。
  - **基础设施**：预留 `hssp-gateway` 配置文件，维护局域网 Nacos（可选）环境。

### 2. 用户业务与数据引擎 (后端B)

- **负责文件夹**：`/hssp-backend/hssp-service` (User/Step子包)
- **核心职责**：
  - **账户体系**：实现用户注册、登录及 Token 颁发。
  - **步数中心**：接收并存储鸿蒙端上传的加密步数数据。
  - **高性能排行**：使用 **Redis ZSet** 实现全站步数实时排行榜（支持分页）。

### 3. 积分逻辑与数据建模 (后端C)

- **负责文件夹**：`/hssp-backend/hssp-model` 与 `/hssp-docs`
- **核心职责**：
  - **对象建模**：维护所有 Entity、DTO、VO，确保三端数据结构统一。
  - **积分业务**：实现步数兑换积分计算、兑换上限校验、积分消费明细。
  - **文档维护**：负责 **Swagger/Apifox** 接口文档的实时更新，作为前端开发的唯一准则。

### 4. 跨平台前端组 (成员D & E)

- **Web端 (D)**：负责 `/hssp-web-admin`。基于 Vue 3 + Element Plus，调用“管理端后端”接口。
- **鸿蒙端 (E)**：负责 `/hssp-harmony-app`。基于 ArkTS，调用“用户端后端”接口，处理传感器数据。

## 二、 四周迭代滚动计划 (Milestones)

### 第一周：架构落位与局域网调通 (Infrastructure)

- **目标**：解决 Mac/Win 混合开发环境下的换行符与 Maven 编译冲突。
- **关键任务**：
  1. 组长发布 `application.yml`，设置 `server.address: 0.0.0.0`。
  2. 后端 C 发布首版 API 文档。
  3. 全员在同一 WiFi 下通过 `http://[组长IP]:8081/hello` 完成链路测试。

### 第二周：功能开发与 Mock 联调 (Feature Injection)

- **目标**：后端逻辑基本闭环，前端完成高保真 UI 模拟。
- **关键任务**：
  1. 后端 B 完成 Redis 排行榜接口；后端 C 完成积分兑换逻辑。
  2. 前端组在代码中使用后端 C 提供的假设数据（Mock），完成所有 UI 交互设计。
  3. 组长完成 JWT 拦截器，确保所有请求开始带 Token 访问。

### 第三阶段：全链路集成测试 (Integration)

- **目标**：移除 Mock 模拟，实现三端局域网真实数据流转。
- **集成场景**：
  1. **场景A**：鸿蒙端上传步数 -> 组长电脑处理 -> 后端 B 更新排行榜 -> 鸿蒙端实时看到排名变动。
  2. **场景B**：Web端调整积分比例 -> 后端 C 计算逻辑自动更新 -> 用户兑换积分结果改变。

### 第四阶段：商业演示与交付准备 (Final Polish)

- **目标**：优化性能，固化文档，准备最终演示。
- **关键任务**：
  1. 整理 `/hssp-docs` 下的部署手册，确保其他电脑可以快速复现环境。
  2. 进行最后一次代码审计，清除所有本地硬编码 IP，改为配置文件引用。
  3. 准备商业演示 PPT，重点突出 **Redis 排行榜** 和 **模块化解耦** 两个技术亮点。

## 三、 协作规范 (Protocol)

1. **Git 规范**：
   - 严禁提交 `target` 文件夹（`.gitignore` 已配置）。
   - Mac 成员严禁修改 Windows 成员提交的 `.bat` 脚本（反之亦然）。
2. **联调规范**：
   - 组长 IP 发生变动时（如 WiFi 重新分配），必须在群组置顶公告。
3. **代码规范**：
   - 所有接口返回必须继承 `hssp-common` 中的 `Result<T>` 类。
   - 数据库字段名统一使用下划线（如 `step_count`），Java 实体类使用驼峰（如 `stepCount`）。
